## Ordering

### CRITICAL

- **"Station Types" appears after "Mock Mode Configuration"** (line 104). Station types are a foundational concept that constrains every subsequent example (`station_type:` on `bench_1.yaml`, capability declarations, validation). They should appear immediately after "Station Fields" / "Instrument Fields" and before any environment-specific example. As written, the reader sees `station_type: voltage_tester` for the first time on line 142 with no scaffolding, then has to scroll back to "Station Types" to understand it. The recipe-progression order is broken.

### WARNING

- **"Capability Declarations" (line 249) duplicates the Station Types section** (line 104). Both explain that `capabilities:` belongs on the type, not the instance. The reader hits the same lesson twice with subtly different wording — merge into the Station Types section, leave a single canonical home for the "capabilities come from catalog refs" rule.
- **"Validation" (line 270) appears mid-flow between "Capability Declarations" and "Shared Instruments"**. Validation is a meta-topic that belongs either right after the field tables (so the reader knows the schema is enforced) or in a closing "Verifying your config" section — not buried between two unrelated configuration topics.
- **"Selecting a fixture at run time" (line 236) is a how-to-run topic, not a how-to-configure topic.** It interrupts the configuration narrative. Either move it to a clearly-labeled "Using your station" section near the end, or drop it and link to a "Running tests" how-to.

### SUGGESTION

- The "Common Configurations" section at the bottom (line 338) repeats what "Multiple Stations" already covered (line 156). Consider folding or cutting — three "Single DMM / Full Production / Production Lab / Development Lab / CI/CD" examples is at least one too many for one page.
- "Best Practices" (line 329) before "Common Configurations" reads backward: a reader scanning the TOC expects examples to support best practices, not follow them. Swap the order, or fold best-practice bullets into the relevant sections inline.

## Voice

### WARNING

- **Imperative how-to voice drops in the descriptive sections.** Lines like "Station configuration is validated by Pydantic models when loaded:" (line 272) and "A station instance (`stations/<id>.yaml`) does not carry a `capabilities:` field — capabilities are derived from each instrument's catalog entry at load time…" (line 251) are explanation-quadrant phrasing. A how-to should say "Don't set `capabilities:` on a station instance — declare them on the station type instead" and show the fix.
- **"Mock values come from `mock_config` in the station, or can be overridden per-test in the sidecar YAML next to your test file"** (line 102) buries the action verb. A how-to phrasing: "To override mock values for a single test, set them in `tests/test_*.yaml` next to the test."
- **Headings mix sentence case and title case.** "Station Fields", "Instrument Fields", "VISA Addresses", "Mock Mode Configuration", "Station Types", "Multiple Stations" use title case; the sub-headings "Running in Mock Mode", "Per-Resource Locking", "Automatic Server Setup" mix the two. "Selecting a fixture at run time" is sentence case. Pick one and apply across the page.
- **"Best Practices" numbered list (lines 329–337) is a wall of one-line tips with no actionable detail.** "Use environment variables — For IP addresses in different environments" doesn't say how (the page never shows env-var interpolation). Either show the mechanism or drop the tip.

### SUGGESTION

- "The `--mock-instruments` flag uses mock instruments instead of real hardware" (line 102) is tautological — "the flag does what its name says." Replace with a sentence the reader doesn't already know.
- "Tests never know the difference" (line 287) is marketing voice in the middle of a config recipe. Reword as a concrete promise: "Test code accesses shared instruments via the same fixture name and call pattern as direct instruments."
- "No special flags are needed." (line 311) — start the sentence with the active idea: "Sharing is automatic — when the fixture defines multiple slots that share a role, the orchestrator …".

## Audience

This page is a how-to, so the audience is a **test engineer writing or editing a station YAML**. Several sections drift into developer/internals territory.

### CRITICAL

- **The "Shared Instruments (Multi-DUT)" section (lines 285–327) names internal classes (`InstrumentServer`, `RemoteInstrumentProxy`) and source paths (`src/litmus/instruments/server.py`).** A test engineer configuring a station does not need to know the class name of the RPC server. This belongs in an architecture / explanation page (linked from here); the how-to should answer only: "What do I put in my station YAML to make this work?" The answer is "nothing — it's automatic," which the reader can absorb in one sentence.

### WARNING

- **"Validation" section (lines 270–283) shows a Python REPL example.** A test engineer doesn't load station YAML from a Python shell — they run `pytest --station=…` or `litmus serve`. Either show how `pytest` surfaces the validation error, or cut the section and add a one-liner "Litmus validates your station YAML at session start; mistakes show up as a clear pytest error before any test runs."
- **"Per [`StationInstrumentConfig`](../reference/models.md) in `src/litmus/models/station.py`"** (line 36). Naming the model file path in a how-to leaks implementation. The reference link is enough; drop the `src/...` path or move it to a "see also" footnote.
- **"Capability Declarations" (lines 249–268) reads like an explanation of the matcher's traversal (`station.instruments → instrument.catalog_ref → catalog/<...>.yaml → capabilities[]`)**, not a recipe. The test engineer's question is "where do I put a capability?" — the page should answer that in one sentence with one example, then link to a deeper explanation.

### SUGGESTION

- "Validates that the fixture's `product_id` / `product_family` matches the active product spec before any test runs" (line 238) — fine technical detail, but the operator-facing version is "Litmus refuses to start the session if the fixture doesn't match the product you selected."
- The "Instrument Types" reference (dmm / psu / scope / eload / funcgen) doesn't appear on this page even though the field table references "Instrument type (dmm, psu, etc.)" — a how-to reader skimming for "what can I put in `type:`?" has to leave the page. Either inline the short table or link to the type list directly from the `type` row.

## Accuracy

### CRITICAL

- **The `name` field is REQUIRED, but the field table says "No" (line 29).** Per `src/litmus/models/station.py:56`, `name: str` has no default — it is required. The page's "Basic Station Config" example includes `name:` correctly, but the field table contradicts the model. Several later examples on this very page (`ci_station.yaml` at line 211, `simple_station` at line 343, `production_station` at line 355, the `bench_1.yaml` example at line 140) omit `name:` and would fail Pydantic validation. Either fix the table (Required: Yes) and add `name:` to the broken examples, or — if you really intended `name` to be optional — fix the model and update the table.
- **The `hostname:` field is undocumented.** `src/litmus/models/station.py:66` defines `hostname: str | None = None`, with a load-bearing role: the session-start resolver auto-matches `socket.gethostname()` so operators can drop `--station=<id>` on the matching bench. This is exactly the kind of "configure once, forget" feature a how-to should surface, and it's missing from both the Station Fields table and every example.

### WARNING

- **Capability id format inconsistency.** The page uses dot-delimited ids like `keysight.34461a.dc_voltage` (lines 118, 263). Confirm against the catalog model whether this matches real catalog ids in the project; the demo catalog and `docs/concepts/stations.md` use the same convention, so this is likely fine, but the page should either link to the canonical id list or note where the reader finds valid capability ids. Today an unknown reader will guess.
- **"Switches → concurrent access (relay operations are atomic)"** (line 295) overstates the source. `src/litmus/instruments/server.py:45` describes a `concurrent_roles` parameter — concurrency is opt-in by role, not an automatic "switches always concurrent." The page implies switches are always concurrent; the code says any role passed in `concurrent_roles` skips the lock. Either document how a station declares `concurrent_roles` (it isn't a YAML field today), or rephrase to "If the orchestrator marks the role as concurrent, access is parallel."
- **"`channels` | No | Dict of named channel→port mappings"** (line 44). The model field is `channels: dict[str, str]` (`src/litmus/models/station.py:32`). The page never shows an example of populating it, and the description "channel→port" is opaque — what is a "port" in this context? Either show one realistic example or remove the row (no example on the page actually uses it).
- **"`supported_phases`" enforcement claim** ("Helps filter / prevents wrong usage" at lines 32, 333). A quick grep shows `supported_phases` is read by `src/litmus/ui/pages/stations/detail.py` for display only — there is no enforcement in the pytest plugin or execution path. Either the page should mark this as "informational only," or the enforcement needs to land in code. Best Practices #3 ("Define supported phases — Prevents wrong usage") is currently false advertising.

### SUGGESTION

- The `--station=stations/bench_1.yaml` form (line 99) and the `--station=bench_1` form (line 242) coexist on the same page. Both work, but be explicit: "`--station=<id>` resolves to `stations/<id>.yaml`; `--station=<path>` works too." A reader copying from one example may not realize they're equivalent.
- "Invalid configurations raise `pydantic.ValidationError` with details about what's wrong" (line 283) — be specific about what the test engineer actually sees. From pytest, what does the error look like? A two-line transcript would land this better than the abstract claim.

## Gaps

### CRITICAL

- **No coverage of the `hostname:` auto-match feature.** This is the single most operator-friendly station feature in the codebase (drop `--station=…` on a bench machine and the right station is picked) and the page never mentions it. New how-to section needed: "Auto-selecting a station by hostname."

### WARNING

- **No "where do station files live?" orientation.** The page jumps into `stations/bench_1.yaml` without telling the reader that station files live in `stations/` relative to the project root, and that `stations/types/` is a sibling directory for templates. The store searches project-local then global (`~/.local/share/litmus/stations/`) per `src/litmus/store.py:425–440` — undocumented here.
- **No example of `litmus discover` for generating a starter station.** `CLAUDE.md` lists `litmus discover` as a first-class command. A how-to on configuring stations should show the bootstrap path (discover → save → edit) before showing hand-authored YAML.
- **`station_type` validation behavior is undocumented.** Per `src/litmus/models/station.py:100` (`validate_station_against_type`), if a station declares `station_type: voltage_tester` but is missing a required role, the resolver returns a human-readable mismatch list. The how-to should show what that error looks like and how to fix it — that's the whole point of declaring a type.
- **No coverage of mock-only stations (`mock: true` without `resource:` or `driver:`).** The CI example at line 211 uses `mock: true` per instrument with no `resource:`/`driver:`, which is legal per the `resource_required_for_real_hardware` validator. But the page never explains the rule. The validator's error message (`src/litmus/models/station.py:37–47`) is the right copy to lift.

### SUGGESTION

- No troubleshooting section. Common station-YAML mistakes: missing `name:` (per the accuracy finding above), `catalog_ref` pointing at a missing catalog entry, `resource:` typo (missing `INSTR` suffix), `station_type:` named but no template file. A "When things go wrong" table at the bottom would save a lot of bench time.
- No mention of how station YAML interacts with `instruments/` asset files (`load_instrument_files` in `src/litmus/store.py:1250`). If a station's `catalog_ref` resolves to an asset record (calibration, asset tag), the reader gets traceability for free — and won't know without being told.
- The CI/CD example uses `mock: true` per instrument; the page never mentions that `--mock-instruments` on the command line is a separate (and stronger) override. Document the precedence: per-instrument `mock: true` vs. CLI `--mock-instruments` vs. profile mock settings.
