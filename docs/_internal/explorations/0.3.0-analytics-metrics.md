# 0.3.0 — analytics / metrics release

**Status:** ROADMAP (2026-06-21). Deferred from 0.2.0 by decision — the metric
*additions* below land in a dedicated 0.3.0 analytics release, designed properly.
Research-backed (sources at the end). Nothing here is built.

## Where we are (0.2.0 baseline)
Litmus ships the full **Tier-1** manufacturing-test set: First Pass Yield, final
yield, failure pareto (by step/measurement), test-time stats, Cpk/Cp, yield trend,
retest rate, time-loss. That's a solid, complete baseline — the gaps are Tier-2
quality-engineering metrics.

## Additions (prioritized)

### Quick wins — SHIPPED IN 0.2.0 (moved out of 0.3.0)
- **Ppk / Pp** — DONE: the existing capability metric computed overall σ (= Ppk's basis)
  but was *mislabeled* "Cpk". Renamed `Cpk/Cp → Ppk/Pp` across the surface. The genuine
  Cpk↔Ppk *distinction* (true Cpk needs within-subgroup σ) is the 0.3.0 item below.
- **RTY (Rolled Throughput Yield)** — 0.2.0 (yield tab): `∏ FPY_step` = `EXP(SUM(LN(step_fpy)))`
  over distinct steps. Defined per-step within a run's step sequence.
- **DPMO / DPPM** — 0.2.0 (yield tab): DPMO = failed_steps / total_steps × 1e6;
  DPPM = failed_runs / total_runs × 1e6.

### Features — new views (own design pass)
- **True Cpk / Cp** (within-subgroup σ) — the genuine short-term capability, distinct
  from the overall-σ Ppk now shipped. For functional-test individuals (one value per
  unit) the within-σ comes from the **moving range** (I-MR: `σ_within = MR̄/1.128`) — so
  this bundles naturally with the SPC work below. Show the Cpk↔Ppk gap as a stability
  signal.
- **Per-measurement SPC control charts** (I-MR / X-bar-R + Western Electric / Nelson
  rules) — plots a measurement in run order with control limits, flags drift *before*
  it crosses spec. Highest-value addition for test engineers ("catch drift before
  yield drops"); in every serious tool (Sciemetric, WATS). Cpk is a snapshot; SPC is
  the movie.
- **Yield cross-tab by station / fixture / operator / shift** — the #1 diagnostic
  drill-down: "is it the *product* or the *fixture*?" A flat FPY can't answer it.
  Needs station/fixture/operator stamped on each run (mostly present).

### Refinements to the existing 6
yield → RTY + DPMO (done 0.2.0); ppk → add true Cpk/Cp (within-σ) + show the Cpk↔Ppk
gap; trend → add per-measurement SPC view; failure_pareto → absolute counts (for DPMO) + a
by-dimension (station/operator/date) sub-view; retest → per-station/fixture (not just
per-serial); time_loss → station utilization % (a lightweight OEE proxy).

## DOCS REQUIREMENT (hard rule — per user, 2026-06-21)
**Every metric added in 0.3.0 must ship with updated docs in the same change:**
- new/refreshed **operator-UI screenshots** for the affected tabs (run
  `scripts/regenerate-ui-screenshots.py`), AND
- a **curated, representative demo dataset** behind them — not the current thin seed.

The current screenshot seed (`examples/07-profiles`, re-seeded by `pytest`) is a poor
analytics demo: it is **all-passing** (so failure_pareto renders an empty state) and
its accumulated data dir carries **stale runs with ~141-hour durations** (garbage
test-time stats). 0.3.0 needs a deliberate demo dataset: real **failures** (for
pareto/DPMO), **realistic durations**, **multiple serials/stations/shifts** (for the
cross-tab), and a **time series** (for trend/SPC). Build that dataset as part of the
analytics work so the screenshots actually show the metrics doing their job.

## Already done (0.2.0, reusable here)
- `litmus.ui.shared.components.format_number()` — `g`-format numeric display (strips
  IEEE float noise). Reuse it for the new metric numbers (Ppk, RTY, DPMO, SPC limits).
- The metric query name `pareto` is already reserved as the forward-looking generic
  name (returns failures by default; the generic `pareto(by=measure)` is part of this
  release — see `read-surface-unification.md` northstar + `query-by-role-name.md`).
