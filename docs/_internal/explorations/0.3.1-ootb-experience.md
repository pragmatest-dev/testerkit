# 0.3.1 — Out-of-the-box (OOTB) experience improvements — execution diary

**Status:** in execution (2026-07-06). Branch `fix/init-headless-and-starter-assets`.
Design contract + progress log for the OOTB batch. Findings here are VERIFIED against
source/empirically this session — do not re-derive.

## Problem

A new user's first loop is `litmus init --starter → pytest → look at results (litmus runs / serve / VSCode)`.
Two gaps hurt that first impression:

1. **Live / just-finished runs are invisible.** `litmus runs` (and the HTTP API + MCP)
   show "No test runs found" during a run and for a beat after it finishes.
2. **The starter is a thin, partly-wrong showcase** — one scalar test, no vectorization,
   a docstring that falsely claimed a "sweep in the sidecar," and an `observe("vin", …)`
   that recorded an *input* into the *outputs* lane.

## Verified findings (do not re-derive)

- **Events are the working baseline.** pytest emits `RunStarted…RunEnded` to the events
  daemon; force-flushed on session close (`EventStore.close` → `EventLog.close` pushes the
  final batch). Flush thresholds (`BufferedIPCWriter`): events 50 rows **/ 1.0 s timer**;
  channels 100 / 1.0 s; live channel sink 0.005 s. `RunEnded` rides the close-flush, not the
  timer — so lowering `flush_interval` does NOT shrink any done-transition.
- **The `runs` view = `runs_materialized ∪ inflight_runs`** (`_runs_duckdb_daemon.py` ~1971).
  The inflight overlay is fed by the runs daemon's `AccumulatorPool` and synced by
  `_overlay_sync_loop` (~50 ms). With a warm daemon a run shows in-flight *only while
  genuinely running* and flips to its outcome the instant it ends — measured: `passed` at
  `pytest_exit − 0.13 s`. **No stale "done shows running" window** in the warm case.
- **ROOT CAUSE A — reader filter.** `RunsQuery.list_recent()` (analysis/runs_query.py:175)
  defaults `include_incomplete=False` → appends `WHERE ended_at IS NOT NULL`, hiding
  in-flight runs. CLI (cli/runs.py:84), API (api/app.py:293), MCP (mcp/tools.py:396,1738)
  all inherit the default. The UI results page already passes `include_incomplete=True`
  (ui/pages/results/list.py:51), so the UI already surfaces live runs.
- **ROOT CAUSE B — no materializer during bare pytest.** The pytest plugin never
  instantiates `RunStore` / `runs_duckdb_manager.acquire` — pytest writes events only.
  Empirically: bare pytest, no reader → `run.materialized` = 0, no runs-daemon process, no
  `data/runs/`, indefinitely; the first *reader* spawns the runs daemon, which then
  materializes via replay. The materializer LOGIC is correct: `_on_event` trigger is
  order-independent (needs `_run_started` AND `_run_ended`); `on_event(replay="unmaterialized_runs")`
  is **gap-free** (cursor = `MAX(event_number)` snapshot *before* replay; watcher started
  before replay covers `> cursor`; replay covers `<= cursor` for unmaterialized runs;
  `_delivered_ids` dedups). So `RunStarted` is never lost regardless of when the daemon
  attaches — the ONLY defect is that nothing launches it during pytest.
- **Sweep params record as inputs automatically.** `run_scope.py:1059` — a pushed vector's
  params become the step's `inputs` (`inputs=coerce_dict(vector.params)`, with `input_units`).
  So `observe("vin", …)` on a swept input duplicates it into the outputs lane (wrong verb +
  wrong lane). `litmus_sweeps` lowers to real `metafunc.parametrize` at **collection**
  (`pytest_generate_tests`, hooks.py:1715) → each vector is a first-class item
  (`test_x[3.3]` …), so it shows in the VSCode Test Explorer.
- **Serial + pytest dep are fine.** Starter `pyproject` sets `--uut-serial=STARTER001` in
  addopts (no prompt). `pytest>=8.0` is a declared *runtime* dependency of `litmus-test`
  (pyproject comment says so), so a Codespace `pip install litmus-test` puts pytest on PATH.

## Decisions (locked)

Live-runs (no blocking at exit — user rejected it; the open filter makes the run visible
continuously so there is no gap to wait on):
1. **Launch the materializer with the session** — `_open_session_for_pytest` acquires the
   runs daemon on the canonical `data_dir` (same shape as the events/channels daemons
   already launched there; session-scoped, NOT per-test/tmp_path → no pids-cgroup hazard).
2. **sessionfinish releases without waiting** — no block-at-exit; daemon materializes async.
3. **Open the read filter** — CLI/API/MCP → `include_incomplete=True`; render in-flight as
   `RUNNING`.
4. **Cold-start reader readiness** — a cold `acquire` waits for the daemon's initial
   attach + `unmaterialized_runs` replay before serving, so a first reader never false-empties.

Starter + guidance:
- **Inline markers, no sidecar** (readability; sidecars are taught in examples/04-05).
- **One honest swept test** `test_output_voltage(context, psu, dmm, verify, vin: float)` with
  `@litmus_sweeps([{"vin":[3.3,5.0,5.5]}])` + `@litmus_limits(characteristic)`, iterating
  `context.connections` (fixture is present → char resolves a connection, matches examples/06).
  No `observe` of an input, no `get_param`, no class.
- **VSCode:** generated `.vscode/settings.json` gets `python.testing.pytestEnabled` +
  `pytestArgs:["tests"]` (no interpreter path → auto-detect local `.venv` or Codespace system Python).
- **AI guidance:** litmus-tests skill gains a "don't `observe` an input" rule.
- **Codespace:** starter repo `postCreateCommand` gains `&& python -m pytest -q` (create-time
  data + smoke test; materializes at build now that the materializer runs with pytest).
