Three changes shipped at 0.3.0:

1. **De-fuse to one row per execution** — detailed in `runs-execution-model.md`
   ("Retry & outcomes — de-fuse to one row per execution"). One-line: every execution is a
   row at its stable `(step_index, vector_index)`; reruns are extra rows counted at ingest;
   drop stored `vector_retry` + the `retry_count`/`has_measurements`/`vector_count` rollups;
   remove `retry_aware_rollup`. **Lossy on migrate** — old v2 fused files can't be un-fused;
   v2 keeps its fused shape + `retry_count`, v3+ get honest per-execution rows.

2. **C5 — instruments → one `instruments` `list<struct>`** (below).

3. **C4 — `uut_serial` → `uut_serial_number`** (below).

## C5 — instruments → one `instruments` `list<struct>` (dense per row)

**Today.** Instruments are the lone N:1 collection still stored as **14 parallel
`step_instruments_*` `list<string>` arrays** (`_row_helpers.py` `INSTRUMENT_ARRAY_KEYS`),
held together by a hand-maintained "same length, same order" positional invariant
(`run_scope.py`). Every other N:1 collection is already `list<struct>` — `measurements`
(reshaped wide→nested in v2), the `inputs`/`outputs` lanes.

**The principle.** An object's fields flatten either way; **cardinality picks the
container.** A 1:1 object flattens to scalar columns (`station_id`/`name`/`hostname`…) — no
alignment risk, exactly one of each. An N:1 collection flattens to *either* parallel arrays
*or* one `list<struct>`. For N:1 the struct is **strictly safer**: it binds each element's
fields together so the dimensionality can't skew, where 14 parallel arrays make "same
length, same order" a hand-maintained invariant — the accidental-misalignment hazard (same
drift class as the daemon's lockstep column lists). The reshape makes the misaligned state
*unrepresentable*. That correctness win — not symmetry — is the reason.

**The name.** An instrument record is a merge: config (`role`, `instrument_id`, `driver`,
`resource`, `protocol`, `catalog_ref`, `calibration`) + **auto-detected identity**
(`manufacturer`/`model`/`serial`/`firmware` from the device `*IDN?` / LXI) + runtime
(`mocked`). It's "the instruments as connected and identified for this run" — neither pure
station config nor pure detection. So the column is bare **`instruments`** (matching
`measurements`/`inputs`/`outputs`), **not** `station_instruments` and not the now-inaccurate
`step_instruments_` prefix.

**Design.**
- **`instruments` `list<struct>`, carried dense on every row** (self-describing, like
  `uut_serial` already repeats). Parquet dict+RLE compresses the repetition to ~nothing —
  verified columnar compression makes the denormalization nearly free, so **no
  cascade/inherit-from-parent machinery**: each row carries its effective set, an override
  is just a different value.
- **Struct fields** = the 14 current ones, with the serial truncation fixed **at birth**:
  `name` (role), `id`, `driver`, `resource`, `protocol`, `manufacturer`, `model`,
  **`serial_number`**, `firmware`, `cal_due`, `cal_last`, `cal_certificate`, `cal_lab`,
  `mocked`. (This absorbs the *instrument* half of the serial rename — there's no separate
  `step_instruments_serial` to rename; those columns are removed.)
- **Derive at ingest into a materialized table — closes a second gap.** Today there is **no
  instruments materialized table**; `/instruments` derives instruments *ad hoc* by scanning
  `step_instruments_id` across history (`ui/pages/instruments/list.py`). So instruments are
  the exception in *two* ways — parallel arrays **and** no projection. C5 fixes both: the
  daemon `UNNEST`s `instruments` into a new materialized **`instruments` table** at ingest
  (grain: **one row per instrument per run** — the inventory flattened), parallel to
  `measurements_materialized` and following the single-source column-list pattern
  (`_INSTRUMENTS_PERSISTED_COLUMNS`). `/instruments` then reads that table via a new Query
  API method instead of scanning arrays — which also brings it onto the Query-API path the
  UI rule requires.
- **Levels.** The `instruments` set is **stored at the run, step, AND vector grains** —
  each row carries its own *effective set*, not a copy of a parent: the **run** row holds the
  full inventory (every instrument connected or used during the run); the **step** row holds
  the instruments used by / available to that step, known from its **fixture parameters /
  reservations**; the **vector** row holds the set the step is using. These are **stored, not
  derived**. Separately, a measurement may name **one** instrument specifically related to it
  via `instrument_name`/`instrument_resource`/`instrument_channel` **inside the measurement
  struct** in the `measurements` array — a per-measurement **usage link**, distinct from (and
  not the source of) the run/step/vector instrument sets.
  - **Implementation status (gap):** the at-rest schema carries `instruments` on all three
    grains, but the *producer* does not yet fill the step/vector grains with their own
    subset — the streaming path stamps the same run inventory (`acc._build_instrument_records()`)
    onto run/step/vector rows, and no per-step instrument-role capture from fixture
    parameters/reservations exists yet (`StepStarted` has no instrument field). Filling the
    step/vector effective sets is follow-on producer work (depends in part on the reservation
    model). `RunScope.set_step_instruments(roles)` is the disconnected skeleton of that
    producer — keep it.
- **Performance-neutral.** Parquet shreds `list<struct>` into the same per-leaf columns as
  the parallel arrays; projection-of-needed-leaves is preserved; `UNNEST` is cleaner than the
  positional zip. The cost is only the breaking reshape → the 3.0 bump.

**Boundary — utilization is explicitly OUT.** C5 delivers the run-scoped *inventory*
(identity + cal traceability) and powers `/instruments`. Instrument **utilization** is a
separate, **event-sourced** follow-on (instruments are also connected in interactive
sessions that emit `fixture.instrument_connected`/`disconnected` but produce no run, so
run-derived utilization would miss all non-run usage). Do not build utilization into C5.

**Consumers to update (~9):** `data/schemas.py` (`_INSTR_ARRAY_TYPES`, the 14 columns →
`instruments` struct), `data/backends/_row_helpers.py` (`INSTRUMENT_ARRAY_KEYS`),
`execution/run_scope.py` (`build_instrument_arrays`/`set_step_instruments` → emit one struct
list), `data/backends/_event_accumulator.py` (`_build_instrument_arrays`),
`data/_runs_duckdb_daemon.py` (new `CREATE TABLE instruments` + `_INSTRUMENTS_PERSISTED_COLUMNS`
+ the `UNNEST` insert), `data/backends/parquet.py`, `analysis/` (new Query API method for the
instruments table), `api/schemas.py`, `ui/shared/services.py` + `ui/pages/instruments/list.py`
(read the materialized table, not array scans).

**Read-time adaptable** (per `schema-versioning-migration.md`): old v2 files' parallel
`step_instruments_*` arrays zip into the struct at ingest — no rewrite needed once the
version-branch exists.

## C4 — `uut_serial` → `uut_serial_number`

**The defect.** `uut_serial` (`schemas.py`) drops "number" off "serial number" while its
bucket-mates keep the noun (`uut_part_number`, `uut_lot_number`). (The *instrument* serial
truncation is handled by C5, whose struct field is born `serial_number`. So C4 is UUT-only.)

**Why the stored name changes, not the CLI.** The standards Litmus targets are **ATML (IEEE
1671) and ASAM**, not OpenHTF/STDF. ATML uses `SerialNumber`/`PartNumber`; IVI/`*IDN?` uses
`SerialNumber`; the LXI discovery reads `<SerialNumber>`. So the *stored contract* carries
the full standards noun. The CLI is a write/input surface (CQRS — write abstracts storage)
and keeps the ergonomic short form, exactly how `--operator` already maps to the
`operator_name` column.

**Shape (clean rename, no alias/shim — pre-release).**
- **Stored column + read surfaces:** `uut_serial` → `uut_serial_number`, across the parquet
  schema (`schemas.py`), the `_accumulator_pool.py` schema copies, the daemon
  tables/tuples/inserts, the `uut_serial` facet (`measurement_facets.py`), and Query-API
  reads. (`uut_part_number`/`uut_lot_number` are already full.)
- **CLI + env unchanged:** `--uut-serial`, `--uut-serials`, `LITMUS_UUT_SERIAL`,
  `LITMUS_UUT_SERIAL_<SLOT>` keep the short form; mapped to `uut_serial_number` at ingest (the
  `run_scope` write boundary, like `--operator`→`operator_name`). *Optional ergonomic:*
  `--uut-serial-number` as an alias to the same dest; `--uut-serial` stays primary.

**Read-time adaptable:** a pure rename → old v2 `uut_serial` projects to `uut_serial_number`
via a column alias at ingest. No rewrite.

## Migration summary

All three ride the one `schema_version` 2.0 → 3.0 bump. Pre-release the migration is regen
(`rm -rf data/`). Designed-for-scale (per `schema-versioning-migration.md`): C4 (rename) and
C5 (array→struct zip) are **read-time adaptable** once ingest branches on the file version;
the de-fuse is **lossy** (old fused files stay fused). So at scale: v2 files coexist with v3
files, adapted at read — not a forced rewrite.
